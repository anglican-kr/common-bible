# ADR-032: 컴포넌트·뷰 층 모듈화 — 동작 스캐폴딩의 단일화

- 일시: 2026-06-06
- 상태: 승인됨 — 구현 대기
- 관련 ADR: ADR-018(app.js 도메인 모듈 분할), ADR-019(ESM 일괄 채택), ADR-013(유닛 테스트 전략), ADR-028(디자인 시스템 — `DESIGN.md` 토큰 사다리), ADR-010(북마크 UI), ADR-022(인용·주석 바텀 시트), ADR-029/030(탭 바·오버레이 패턴)

## 맥락

비주얼 디자인 층은 이미 성숙하다 — ADR-028 이 색·간격·타이포·반경·elevation·모션을
`DESIGN.md` + `:root` 토큰 사다리로 단일화했고, 토큰 전면 채택까지 끝났다. 그러나 그
토큰을 입는 **클라이언트 JS 의 뷰 구현은 제각각**이다. 문제의 핵심은 마크업(겉모습)이
아니라 **동작(behavior) 스캐폴딩의 복제**다. 같은 생명주기 코드를 모듈마다 손으로 다시
짜고 있어, 한 번 고친 버그가 다른 사본에서 되살아나고 파일이 비대해진다.

### 증상 1 — 오버레이 생명주기가 최악의 중복원

`trapFocus` + `setInert` + Escape 리스너 + 열기/닫기 + 포커스 복원의 한 묶음을, 오버레이가
뜨는 곳마다 따로 재구현하고 있다. 실측(`grep`)으로 `trapFocus` 호출 지점만 **12곳**:

| 파일 | 오버레이 | trapFocus 위치 |
|---|---|---|
| `bookmark.js`(2,957줄) | 드라이브 연결 해제 모달 | `:823` |
| | 북마크 드로어(바텀 시트) | `:1034` |
| | 저장 모달 | `:2112` |
| | 새 폴더 모달 | `:2130` |
| | 병합 모달 | `:2234` |
| | 확인 모달 | `:2296` |
| | 장 삭제 선택 모달 | `:2374` |
| | 가져오기 모달 | `:2500` |
| `install.js` | 설치 안내 모달 | `:426` |
| `citations.js`(1,140줄) | 인용 본문 바텀 시트 | `:966` |
| `settings-ui.js` | 설정 팝오버 | `:800` |
| `views-routing.js`(2,324줄) | 읽기 헤더 팝오버 | `:482` |

즉 **모달 7 + 드로어 1 + 시트 1 + 팝오버 2 + 설치 모달 1**. 각자 포커스 트랩 cleanup
핸들(`_bmSaveModalTrap`, `_driveDisconnectTrap`, `installModalTrap`, `_sheetTrapCleanup`,
`cleanupTrap` …)을 모듈 스코프 변수로 들고, 열 때 직전 포커스를 저장했다가 닫을 때 복원하는
배관을 반복한다. 게다가 **일관성마저 깨져 있다** — 배경 비활성화(`setInert`)는
`bookmark.js`·`install.js` 두 곳만 래핑해 쓰고(`setBookmarkBackgroundInert`,
`setBackgroundInert`), 팝오버·시트 경로는 `setInert` 를 거치지 않는다. 이것이
`bookmark.js`·`views-routing.js` 비대화의 주된 원인이며, 접근성(WCAG 2.1 AA) 회귀가
한 사본에서만 슬그머니 들어올 위험을 키운다.

### 증상 2 — 바텀 시트 드래그-리사이즈 배관 중복

`bookmark.js` 드로어와 `citations.js` 시트가 각각 pointerdown/move/up + 높이 클램프 +
스냅 애니메이션을 따로 구현한다. 공유되는 것은 순수 판정 함수 `dragReleaseAction(h, vh)`
(close/snap-min/stay, `helpers.js:137`) **하나뿐**이고 — 이 함수 자체가 "두 사본이
임계값을 따로 두다 close 분기가 영영 도달 불가였던" Bugbot 회귀에서 태어났다(주석 참조).
즉 포인터 배관은 여전히 두 벌이라 같은 클래스의 버그가 다시 생길 표면이 남아 있다.

### 증상 3 — 빈 상태 마크업 중복

`bm-empty`(`bookmark.js:1626`, 아이콘 + 제목 + 안내)와 `search-empty-state`
(`search.js:341`)이 사실상 같은 구조(아이콘·제목·부제)를 별도 클래스·별도 코드로 짠다.
CSS 도 `.bm-empty*` 와 `.search-empty-state` 로 갈라져 있다(`style.css:1667`, `:4718`).

### 이미 있는 것 — 프리미티브는 있고, 컴포넌트 추상화만 없다

`js/app/helpers.js`(`window.appHelpers`)에 저수준 프리미티브가 모여 있다: `el`(태그+속성+
자식으로 DOM 생성), `clearNode`, `setInert`, `trapFocus`, `dragReleaseAction`. 빠진 것은
이 위에 얹는 **컴포넌트 단위 추상화** — 생명주기를 한 곳에 가두는 컨트롤러와, 반복되는
표현 조각을 순수 함수로 만드는 빌더 — 뿐이다. 프레임워크가 필요한 게 아니라, `el()` 바로
위 얇은 층이 비어 있는 것이다.

## 결정

`el()` 위에 **얇은 바닐라 컴포넌트 층**을 도입한다. 프레임워크 도입이 아니라, 기존
ADR-018(IIFE + `window` 네임스페이스) / ADR-019(ESM) 패턴을 그대로 따르는 **무의존성**
모듈이다. 비주얼은 계속 `DESIGN.md` 토큰을 입고, 이 층은 **동작과 구조의 재사용**만 맡는다.

### 1. overlay/dialog 컨트롤러 팩토리 — 생명주기를 한 곳에 가둔다 (최대 이득)

열기/닫기/Escape/`inert`/포커스 트랩/포커스 복원의 한 묶음을 단일 팩토리로 가둔다. 대략의
계약(세부는 구현 시 확정):

```
const ctl = createOverlay(el, {
  onOpen?, onClose?,          // 생명주기 훅
  inertSelectors?,           // 배경 비활성화 대상 (setInert 위임)
  initialFocus?, returnFocus?,// 열 때 포커스 이동 / 닫을 때 복원(기본 = 직전 활성요소)
  closeOnEsc = true,
  closeOnBackdrop?,
});
ctl.open(); ctl.close(); ctl.isOpen;
```

내부에서 `trapFocus`·`setInert` 를 호출하고 cleanup 을 자기 안에서 관리하므로, 모듈마다
흩어진 `_*Trap` 핸들·직전 포커스 저장 변수·Escape 리스너 등록/해제가 사라진다. 모달 7 +
드로어 + 시트 + 팝오버 2 + 설치 모달이 **모두 이 한 컨트롤러를 공유**한다. 부수 효과로
`setInert`(배경 비활성화)가 **모든** 오버레이에 일관 적용돼 접근성 표면이 균일해진다.

### 2. sheet 팩토리 — overlay 컨트롤러 + dragReleaseAction 결합

(1) 위에 드래그-리사이즈 바텀 시트의 포인터 배관(pointerdown/move/up, 높이 클램프, 스냅
애니메이션)을 한 번 구현하고 `dragReleaseAction` 의 close/snap-min/stay 판정을 붙인다.
`bookmark.js` 드로어와 `citations.js` 시트가 이 팩토리를 공유 — 순수 판정 함수만이 아니라
**배관까지** 단일화돼 증상 2 의 버그 표면이 사라진다.

### 3. 표현용 빌더 — 순수 함수로 DOM 반환(유닛 테스트 용이)

부수 효과 없이 인자 → DOM 노드를 반환하는 작은 빌더들. DOM-only 라 ADR-013 유닛으로 직접
검증 가능(JSDOM/vm 하네스):

- `button({ variant, label, onClick, … })` — `DESIGN.md` 컨트롤 토큰을 입은 버튼 변형.
- `iconButton({ icon, label, … })` — 44px 터치타깃 아이콘 버튼(헤더·바 공용).
- `emptyState({ icon, title, subtitle })` — 증상 3 의 `bm-empty`/`search-empty-state` 통합.
- 리스트 행 빌더 — 북마크·검색 결과 등 반복 행의 공통 골격.

빌더는 **구조와 토큰 클래스만** 책임지고, 데이터 바인딩·이벤트 배선은 호출부가 유지한다
(과도한 추상화 회피).

### 점진 교체 — 한 곳씩 갈아끼움

새 층을 만든 뒤 **한 오버레이씩** 옮긴다(한꺼번에 갈아엎지 않고 곁에 새 컨트롤러를
세워 하나씩 갈아끼우며, 매 단계 앱이 깨지지 않게 유지). 각 단계마다 e2e(ADR-013 분담: DOM-heavy·라우팅은
e2e 책임) 통과를 유지하고, 옮긴 빌더는 유닛을 추가한다. 한 번에 전부 갈아엎지 않는다 —
오버레이 컨트롤러를 먼저 세우고(최대 중복 제거), 모달→드로어→시트→팝오버 순으로 흡수한 뒤
빌더로 표현 조각을 정리한다. **보일러플레이트 생성기(스캐폴드 CLI 등)는 컴포넌트 계약이
실사용으로 안정화된 이후**에만 검토한다(계약이 흔들리는 동안 생성기는 부채만 늘린다).

## 검토한 대안

- **(a) 프레임워크 도입 (Lit / Preact / Web Components 등)** — 컴포넌트 모델·반응성·생명주기를
  공짜로 얻지만, ADR-001(빌드 단계 0 Vanilla SPA)·ADR-019(빌드 0 ESM) 의 핵심 전제를
  깬다. 번들러·런타임 의존·학습 곡선·SW 프리캐시 목록 변경이 따라오고, 우리가 겪는 문제는
  "반응성 부재"가 아니라 "생명주기 코드 복제"라 도구가 과하다. 무의존성 원칙(이 프로젝트의
  배포·캐시 단순성의 뿌리)을 지키는 편이 비용 대비 이득이 크다. **기각.**
- **(b) 현행 유지(아무것도 안 함)** — `bookmark.js`(2,957)·`views-routing.js`(2,324)
  비대화가 계속되고, 접근성·드래그 버그가 사본마다 따로 회귀한다(증상 2 의 Bugbot 사례가
  이미 발생함). 노트·캘린더 등 후속 기능(ADR-026 계열)이 같은 오버레이 패턴을 또 한 벌
  복제하게 된다. **기각.**
- **(c) 마크업만 헬퍼화** — `el()` 위에 표현 빌더(증상 3)만 추가하고 생명주기는 손대지
  않는 안. 가장 싸지만 **최대 비용원인 오버레이 생명주기(증상 1·2)를 그대로 둔다**. 빌더는
  결정 (3) 으로 이미 포함하되, 그것만으로는 부족하다는 게 조사 결론. 부분 채택(전체로는 기각).

## 영향

- **코드량·가독성**: `bookmark.js`·`views-routing.js`·`citations.js`·`install.js`의
  생명주기 보일러플레이트가 컨트롤러로 흡수돼 줄고, 도메인 로직만 남아 읽기 쉬워진다.
- **접근성**: `inert`·포커스 트랩·포커스 복원이 모든 오버레이에 일관 적용돼 WCAG 2.1 AA
  회귀 표면이 단일 지점으로 수렴.
- **테스트(ADR-013)**: 빌더는 순수 함수라 유닛 케이스가 늘고, 오버레이/시트 와이어링은
  계속 e2e 가 책임(역할 분담 유지).
- **디자인 정합(ADR-028)**: 빌더가 `DESIGN.md` 토큰 클래스를 한곳에서 입혀, 토큰 이탈이
  생길 표면이 줄어든다(컴포넌트 = 토큰 채택의 단일 통로).
- **후속 기능**: 노트·캘린더(ADR-026 계열) 등이 새 오버레이를 공유 컨트롤러로 바로 구현.
- **위험**: 점진 교체 중 신·구 패턴 공존 구간이 생긴다 — e2e 게이트로 단계마다
  회귀를 막고, 한 번에 한 오버레이만 옮겨 위험을 좁힌다. 빌더 과추상화는 "구조·토큰만,
  배선은 호출부" 원칙으로 억제.
