# ADR-005: 검색 인덱싱 전략

- 일시: 2026-03-29
- 상태: 승인됨

## 맥락

성경 전체(35,482절)를 대상으로 전역 전문 검색을 구현해야 한다. 프레임워크 없는 Vanilla JS SPA 환경에서, 외부 서버 없이 클라이언트 측에서 모든 검색이 이루어져야 한다.

고려 사항:
- 코퍼스 규모: 35,482절, 약 6.5MB (클린 텍스트 기준)
- 오프라인 동작 필수 (PWA)
- 외부 라이브러리 의존성 최소화 (프로젝트 원칙)
- 한국어 텍스트 특성: 형태소 분석 없이도 부분 문자열 매칭이 유의미

## 검토한 대안

### A안: 외부 검색 라이브러리 (lunr.js, Fuse.js, FlexSearch 등)

- 장점: 성숙한 토크나이저, 퍼지 매칭, 가중치 랭킹
- 단점:
  - 한국어 형태소 분석 미지원 또는 별도 플러그인 필요
  - 번들 크기 증가 (lunr ~8KB, Fuse ~15KB, FlexSearch ~6KB)
  - 라이브러리 버전 관리 부담 (CDN 의존 또는 vendoring)
  - 인덱스 포맷이 라이브러리에 종속

### B안: 빌드 시 역색인(Inverted Index) 생성

- 장점: 검색 시 O(1) 토큰 룩업, 응답 빠름
- 단점:
  - 한국어 토크나이저 구현 필요 (형태소 분석기 or n-gram)
  - 형태소 분석기 없이 n-gram 사용 시 인덱스 크기 폭증 (2-gram 기준 ~20MB 이상 추정)
  - 부분 문자열 검색("빌라도" → "빌라도에게")에 n-gram 크기 선택이 까다로움
  - 구현 및 유지보수 복잡도 높음

### C안: 플랫 JSON 배열 + 선형 스캔 (채택)

- 장점:
  - 구현 극히 단순 (`String.includes` 한 줄)
  - 인덱스 포맷이 직관적이고 라이브러리 비종속
  - 한국어 부분 문자열 매칭에 자연스럽게 대응 (형태소 분석 불필요)
  - 인덱스 크기 = 원본 텍스트 크기 (추가 팽창 없음)
  - Web Worker에서 실행하므로 메인 스레드 블로킹 없음
- 단점:
  - O(n) 검색 시간 (35,482절 전체 순회)
  - 랭킹/관련도 정렬 미지원 (출현 순서 = 성경 순서)

## 결정

**C안 채택**: 빌드 시 Python 스크립트(`search_indexer.py`)로 플랫 JSON 배열을 생성하고, 런타임에 Web Worker에서 선형 스캔으로 검색한다.

### 근거

1. **코퍼스 규모가 충분히 작다**: 35,482건의 `String.includes` 선형 스캔은 현대 기기에서 수십 ms 이내에 완료된다. Web Worker에서 실행하므로 UI 블로킹도 없다.
2. **한국어 특성**: 한국어는 교착어로 조사가 붙기 때문에 ("빌라도" → "빌라도에게", "빌라도가") 부분 문자열 매칭이 역색인보다 오히려 자연스럽다. 역색인은 형태소 분석 없이는 이런 변형을 놓친다.
3. **의존성 제로**: 프로젝트 원칙(서버 의존성 없음, 외부 라이브러리 최소화)에 부합.
4. **유지보수 단순성**: 인덱스 포맷이 `{b, c, v, t}` 4개 필드의 배열이라 디버깅과 수정이 용이.

### 인덱스 구조

```json
{
  "meta": {
    "aliases": { "창": "gen", "창세": "gen", "창세기": "gen", ... },
    "books": { "gen": { "ko": "창세기", "bo": 0 }, ... }
  },
  "verses": [
    { "b": "gen", "c": 1, "v": 1, "t": "한처음에 하느님께서 하늘과 땅을 지어내셨다." }
  ]
}
```

- `meta.aliases` (301개): 절 참조 검색("창세 1:3") 시 한국어 약어 → bookId 변환
- `meta.books`: 정렬 인덱스(`bo`) + 한국어 이름. 결과를 성경 순서로 표시하기 위해 사용
- `verses[].t`: `¶` 제거, `\n` → 공백 치환한 클린 텍스트. 프롤로그(집회서 머리말) 제외

### 성능 한계 및 향후 대응

코퍼스가 기도서·성무일과 등으로 확장되어 10만 건 이상이 되면 선형 스캔의 지연이 체감될 수 있다. 그 시점에 B안(역색인) 또는 `Intl.Segmenter` 기반 토크나이저 도입을 재검토한다.
