# ADR-026: 메뉴 기반 라우팅 전환 + 노트 기능 + 하단 내비게이션

- 일시: 2026-05-30
- 상태: 승인됨 — 구현 중 (Stage 0~5, release/1.6.0)
- 관련 ADR: ADR-009(History API 라우팅), ADR-010(북마크 모델·UI), ADR-011(Drive 동기화),
  ADR-005(검색 인덱싱), ADR-014(검색 이력), ADR-018(모듈 분할), ADR-020(저장소 토폴로지),
  ADR-023(설정 토글), ADR-024(목록 탭·헤더 내비), ADR-025(헤더 elevation)

## 맥락

앱이 커지면서 주요 영역마다 **상호작용 모델이 제각각**이 됐다.

- **읽기**: History API 라우트 (`/`, `/old_testament`, `/john/3` …)
- **검색**: 데스크탑은 헤더 인라인, 모바일은 FAB → 바텀시트(`#search-sheet`, 드래그/스크림)
- **북마크**: 우측 드로어(`#bookmark-drawer`, 드래그/리사이즈/스크림/스와이프 행)
- **설정**: 기어 → 팝오버(`#settings`, 위치 계산/포커스트랩)

같은 앱 안에서 라우트·시트·드로어·팝오버가 뒤섞여 있어, 뒤로가기·딥링크·z-index·포커스
관리가 영역마다 다르게 동작했다. 또 신규 기능인 **노트**는 자유 마크다운 + 구절 인용을
담는 장문 콘텐츠라, 시트나 팝오버에 욱여넣기 어렵고 독립된 화면이 필요하다.

→ 모바일을 **네이티브 탭바 앱처럼 "모든 주요 영역이 풀스크린 라우트"** 로 통일하고, 하단
메뉴가 그 화면들을 오간다. 데스크탑은 넓은 화면 이점(본문+참조 동시 보기)을 살리기 위해
당분간 기존 오버레이를 유지한다(모바일 우선, 데스크탑 사이드패널화는 후속).

## 결정

### 1. 메뉴 기반 라우팅 전환 (모바일)

모바일(≤768px)의 검색 시트·북마크 드로어·설정 팝오버를 **풀스크린 라우트**로 이관한다.

| 라우트 | 화면 | 기존(모바일) |
|---|---|---|
| `/` · `/old_testament` … | 읽기(목록·장·본문) | 라우트 (불변) |
| `/search` | 검색 | 시트 → 화면 |
| `/bookmarks` | 북마크 | 드로어 → 화면 |
| `/notes` · `/notes/:id` | 노트 목록/에디터 | 신규 |
| `/settings` | 설정 | 팝오버 → 화면 |

데스크탑(≥769px)은 현행 유지 — 헤더 검색바, 설정 팝오버, 북마크 드로어. 같은 라우트가
해상도에 따라 다르게 렌더되는 반응형 분기는 두지 않고, 모바일에서만 라우트 셸을 쓴다.

### 2. 하단 내비게이션 4탭 (모바일 전용)

`#bottom-nav` — **읽기 / 검색 / 북마크 / 노트** 4개 탭. `position: fixed; bottom: 0`,
`env(safe-area-inset-bottom)` 반영. 데스크탑에선 `display: none`.

- 설정은 탭에 넣지 않는다 — 헤더 기어 진입 유지(향후 설정 항목이 늘어날 것을 고려해
  독립 `/settings` 화면으로). 탭은 자주 쓰는 4개로 제한해 엄지 도달과 균형을 확보.
- 활성 탭은 현재 라우트에서 파생(`routeToTab()`), `aria-current="page"`.
- `#app` 에 하단 패딩(탭 높이 + safe-area)을 더해 콘텐츠가 가리지 않게 한다. 검색 FAB는
  모바일에서 은퇴(Stage 2).

#### 2.1 오디오 바 공존 — 미니플레이어 스택

하단에 바가 둘(항상 보이는 내비 + 재생 시 오디오 바)이 되므로 **미니플레이어 패턴**으로
공존시킨다(Spotify/팟캐스트 톤).

- 내비는 화면 맨 아래 고정. **오디오 바는 그 바로 위에 슬림 스트립**으로 핀
  (`bottom: var(--bottom-nav-height)`). 재생 중에도 내비는 항상 접근 가능.
- **질감**: 내비는 솔리드(`var(--bg)`), 오디오 바는 기존 frosted glass 유지. 오디오 바는
  본문 위를 덮으며 떠 있고 내비는 바닥에 앉는 역할 분리. (ADR-025의 "위 헤더는 솔리드"와
  대칭 — 바닥 내비도 솔리드, frosted 는 떠 있는 오디오에만.)
- **공간**: 작은 폰에서 내비+오디오가 바닥을 많이 먹으므로, **읽기 화면에서 스크롤을 내릴
  때 내비를 숨기고(transform: translateY(100%)) 올릴 때 다시 표시**(auto-hide)해 읽기
  공간을 회수한다. 목록·검색·노트·설정 화면에서는 내비 항상 표시. `prefers-reduced-motion`
  존중. 내비 상단엔 본문과 분리하는 elevation(보더/그림자, ADR-025 바닥 버전).
- 데스크탑은 내비가 없으므로 오디오 바는 현행 그대로(`sticky; bottom: 0`).

### 3. 탭별 마지막 위치 복원

각 탭은 **마지막으로 머문 라우트**를 인메모리로 기억하고(`_tabLast[tab]`), 다른 탭에서
그 탭을 누르면 그 위치로 복원한다. 이미 해당 탭에 있으면 탭 루트로 보낸다.

- 완전한 탭별 N-스택(탭 안 뒤/앞 기록 분리)은 브라우저 단일 History와 충돌하므로 두지
  않는다. "돌아오면 쓰던 페이지로"는 **페이지 단위 마지막 위치 복원**으로 충족한다.
- 재시작 지속(sessionStorage/localStorage)은 두지 않는다 — 세션 내 인메모리만. "읽기"는
  어차피 읽기 위치가 영구 저장돼(ADR 기존) 재시작에도 복원된다.

### 4. 노트 — Drive 우선 장문 마크다운

#### 4.1 저장 모델 — Drive가 진실의 원천

노트는 사용자가 직접 쓴 장문이라 **유실이 치명적**이다. iOS PWA는 7일 미사용 시 스토리지
축출 정책이 있고 `navigator.storage.persist()` 도 보장되지 않으므로, 북마크의 로컬 우선
모델을 그대로 쓰면 위험하다. 따라서:

- **Google Drive = 진실의 원천. 노트는 Drive 연결을 전제로 한다.** 미연결 시 노트 탭은
  연결 유도 게이트를 보여준다(북마크는 기존대로 로컬 우선 — 노트만 Drive 게이트).
- **별도 Drive 파일 `notes.json`**(appDataFolder)에 저장한다. 북마크·설정이 든
  `sync.json`(ADR-011)과 분리 — 노트는 더 크고 Drive 우선 쓰기 패턴이며 export/import
  대상이라, 한 파일에 묶으면 노트 편집마다 북마크 전체를 재업로드하게 된다.
- **IndexedDB = 폐기 가능한 오프라인 캐시 + 미전송 큐.** iOS가 캐시를 축출해도 Drive에서
  다시 받으면 되므로 무손실(캐시는 버려도 되는 사본). 오프라인 편집은 큐에 쌓아 재연결 시
  flush 하고, 미전송 상태를 UI에 표시한다. `persist()` 는 계속 시도한다.

#### 4.2 노트 데이터 모델

```
Note {
  id: string;
  type: "note";
  title: string;            // 비면 본문 첫 줄에서 파생
  body: string;             // 마크다운 원본
  date: number;             // 노트가 다루는 날짜(편집 가능, 기본 = 작성일). 캘린더 키.
  refs?: { bookId, chapter, verseSpec }[];  // 연결된 구절(인용 출처)
  createdAt: number;
  updatedAt: number;        // 동기화 LWW 키(_u에 매핑)
}
```

- 평면 목록(폴더 없음), 기본 정렬은 `date` 내림차순.
- `date` 는 작성일과 분리된 편집 가능 필드 — "오늘 어제 묵상을 기록" 같은 일기형 사용과
  장기 로드맵(성무일과/교회력)을 위해 도입.

#### 4.3 노트 화면 — 목록 ↔ 캘린더 토글

노트 탭은 **목록 뷰**와 **캘린더 뷰**를 사용자가 토글하며, 선호는 설정으로 저장(동기화).

- 캘린더는 순수 JS 월 그리드(무라이브러리). 이전/다음 달, 오늘 강조, **노트가 있는 날짜
  셀에 점 마커**(`date` 기준). 날짜 탭 → 그날 노트 목록. `role=grid` + 셀 aria-label에
  노트 수. `prefers-reduced-motion` 존중.

#### 4.4 마크다운 엔진 (경량 자체 파서)

외부 라이브러리 없이(프로젝트 원칙·ADR-001) 작은 자체 파서를 `js/app/markdown.js` 로 둔다.

- 지원: 제목(`#`~`###`), 굵게/기울임, 순서/비순서 목록, 인용(`>`), 링크, 수평선,
  인라인 코드, 체크박스(`- [ ]`).
- **보안(필수)**: 원시 HTML 패스스루 금지(`< > &` 이스케이프), 생성 태그만 출력. 링크는
  `http(s):` 와 내부 경로(`/…`)만 허용하고 외부 링크엔 `rel="noopener"`. CSP가 인라인
  스타일을 막으므로 렌더 결과에 `style=` 속성을 쓰지 않고 클래스만 사용.
- 도구모음: 굵게·기울임·제목·목록·인용·링크. 선택 영역을 마크업으로 감싸고 커서/선택을
  보존한다. **편집/미리보기 토글**(미리보기 시 키보드 내림).

#### 4.5 진입점 · 구절 가져오기

- 하단바 "노트" → 목록/캘린더.
- **절 선택 액션바(`#verse-select-bar`)에 "노트" 버튼** 추가 → 선택한 절 본문을 인용으로
  단 새 노트를 만들고 에디터로 이동.
- 에디터 **"북마크 삽입" 피커** → 저장된 북마크 선택 시 그 구절 본문을 인용블록(`>`)으로
  커서 위치에 삽입 + 출처 링크(`[창세 1:1-3](/genesis/1/1-3)`). 본문은 고정 번역이라
  스냅샷(박제)으로 안전.

#### 4.6 백업 — 파일 내보내기/가져오기

- **JSON 풀백업**: 전 노트를 한 JSON으로 내보내고 가져온다(병합/덮어쓰기). 북마크 export
  패턴과 일관(ADR-010). Drive 우선이라도 사용자 소유의 오프-드라이브 백업이 된다.
- **개별 노트 `.md` 내보내기**: 노트 하나를 마크다운 파일로 저장(다른 앱 호환).

### 5. 연합 검색 (federated)

노트는 동적 사용자 데이터라 빌드 시 정적 인덱스(ADR-005)에 넣을 수 없다.

- **성경**: 기존 Web Worker + 정적 인덱스(불변).
- **노트**: 메인 스레드에서 노트 스토어를 즉석 선형 스캔(N이 작음). 제목 + 평문화한 본문
  + 연결 구절 라벨을 대상으로 매칭. 노트별 소문자 평문을 캐시.
- **소스 스코프 필터**: 검색 UI에 `[전체] [성경] [노트]` 세그먼트(기존 `#search-sheet-chips`
  툴바 자리). "전체"는 *내 노트* 섹션을 위에, *성경*을 아래에 그룹으로. 성경의 `in:`
  연산자(ADR-005)는 유지. 두 검색은 병렬로 돌고 `searchId` 로 경쟁 상태를 막는다.
- 검색 이력(ADR-014)은 같은 입력창 공유, 로컬 전용 유지.

### 6. 단계적 이관 (release/1.6.0)

한 release 브랜치로 통합하되, 영역별 feature 브랜치 → `release/1.6.0` PR 로 나눠 진행한다.

- **Stage 0**: 라우팅 셸 + 하단 4탭 + 탭별 위치 복원 + `#app` 패딩·오디오바 스택.
- **Stage 1a**: 노트 코어(모델·IDB 캐시/큐·`notes.json` transport·머지·Drive 게이트·목록/
  에디터 화면·마크다운 엔진+도구모음·절선택 진입·북마크 삽입).
- **Stage 1b**: 캘린더 뷰 + 백업(JSON import/export, 개별 `.md`).
- **Stage 2**: 검색 화면화(모바일) + 연합 노트 검색 + 스코프 필터.
- **Stage 3**: 북마크 화면화(드로어 → `/bookmarks`).
- **Stage 4**: 설정 화면화(`/settings`).
- **Stage 5**: 모바일 FAB/시트/드로어/팝오버 데드패스 은퇴 + 관련 ADR 개정 + 유닛 테스트 +
  1.6.0 릴리스 범프.

각 Stage 종료 시 본 ADR과 관련 ADR(024/025/010/011/020/005)의 상태·개정 블록을 갱신한다.

## 대안

- **데스크탑도 풀스크린 라우트로 통일**: 가장 단순하나 넓은 화면의 "본문+참조 동시 보기"
  이점을 잃어 기각(모바일 우선, 데스크탑 사이드패널화는 후속 검토).
- **노트를 기존 `sync.json` 에 묶기**: 머지/전송 인프라 재사용은 되나, Drive 우선·장문·
  export 대상인 노트가 로컬 우선 북마크와 한 파일에 섞여 편집마다 전체 재업로드를 유발.
  별도 `notes.json` 로 분리.
- **노트도 로컬 우선(Drive 선택)**: 북마크와 일관되나 iOS 축출로 장문 유실 위험. Drive
  우선 + 폐기 가능 캐시로 결정.
- **완전한 탭별 N-스택**: 브라우저 단일 History와 충돌하고 복잡도가 큼. 마지막 위치 복원으로
  요구를 충족.
- **마크다운 외부 라이브러리(marked/markdown-it)**: 번들/CSP 검토 부담 + 무의존 원칙 위배.
  경량 자체 파서.

## 개정 (2026-05-31) — 노트 저장: 단일 notes.json → per-note 파일 + 인덱스, updatedAt+hash, 내구성 (Stage 1a-①)

§4.1 의 "별도 Drive 파일 `notes.json`" 을 다음으로 대체한다. 노트는 장문이라 콘텐츠
해시·바이트·행 단위 묶음이 맞지 않고, **수정 시각**과 **노트 단위 분리**가 더 중요하다.

> **Drive 저장 = per-note 파일 + 가벼운 인덱스 (§4.1 의 단일 notes.json 대체).**
> appDataFolder 에 인덱스 1개 + 노트별 파일 N개로 둔다.
> - `notes-index.json` — `{ schemaVersion:1, deviceId, notes:{ <id>: {title,date,createdAt,updatedAt,hash} }, tombstones:{ <id>: ts } }`. 본문 없음(작음). 목록·캘린더 렌더와 변경 감지는 인덱스만으로.
> - `note-<id>.json` — 노트 1개 전체. 노트 1개 편집 시 그 파일 + 작은 인덱스만 업로드(다른 본문 불변).

> **변경 감지 = `updatedAt`(편집 시각) + 콘텐츠 `hash` 조합.** 크기/바이트 대조는 쓰지
> 않는다. `updatedAt` 은 **내용이 실제로 바뀔 때만** 갱신(열람·무변경 자동저장은 갱신 안
> 함). `hash`(정규화된 title+body+date+refs, 가벼운 동기 cyrb53)는 **동일성/변경 판정**,
> `updatedAt` 은 진짜 충돌 시 **LWW 순서**. 노트별 `syncedHash`/`syncedUpdatedAt` 워터마크로
> 더티 판정. 둘 다 변경됐어도 hash 가 같으면 충돌 아님(수렴); 다르면 LWW 승자 + 패자를
> "제목 (사본)" 으로 새 id 복제(글 유실 방지). 순수 `planSync`/`pickConflictWinner` 로
> 유닛 테스트(`tests/unit/notes-store.test.js`).

> **해시 비용 최소화 — 자동저장에서 분리.** 자동저장은 IDB 저장 + dirty 표시만(해시 X).
> 해시는 **동기화 시점에 더티 노트에 1회**만. → 해시 횟수 = 동기화 빈도(키 입력과 무관).

> **로컬 = 폐기 가능 IDB 캐시(`common-bible-notes`).** 스토어 `notes`/`tombstones`/`synced`/
> `meta`. iOS 가 캐시를 축출해도 Drive 인덱스+파일에서 복원(무손실).

> **내구성 4겹 (앱/브라우저 종료·OS 메모리 회수 대비).** ①짧은 디바운스+blur → IDB,
> ②`visibilitychange→hidden`·`pagehide` 시 활성 에디터 버퍼 즉시 flush(iOS 에선
> `beforeunload` 불신), ③같은 핸들러에서 **동기 localStorage 드래프트** 1건(IDB 비동기 쓰기
> 완료 전 종료 대비 보증), ④재실행 시 드래프트가 더 최신이면 복원. **생명주기/드래프트/복원은
> `notes-store` 가 소유**하고, 에디터는 `beginEditing(id, getBuffer)`/`endEditing()` 로 버퍼
> 게터만 등록한다(lifecycle 이벤트는 안 다룸).

> **동기화 연동 = B′ 하이브리드.** 노트는 자체 재시도 루프를 두지 않고 **북마크 FSM 사이클에
> 올라탄다**: `state-machine.js` 가 sync.json 사이클 성공(SYNC_DONE) 직후 `window.notesStore?.
> syncWithToken(_token)` 훅 한 줄로 노트 왕복(인덱스 대조 + 바뀐 노트 파일만)을 fire-and-forget
> 호출. 무거운 인증·온라인·재시도 상태는 FSM 한 곳, notes-store 는 UI 용 가벼운 status
> (`synced/pending/syncing/offline/conflict/error`, 단일 `_setStatus`)만 소유. `transport.js`
> 의 파일 ops 는 파일명 파라미터화(`findFileId`/`listFiles`/`downloadFile`/`uploadFile`/
> `deleteFile`)하고 기존 `*SyncFile` 은 wrapper 로 유지.

이로써 §8 "영향" 의 `transport.js(notes.json 파일 ops)" 는 per-note + 인덱스로, 유닛 테스트의
"노트 스토어 머지" 는 `notes-store.test.js`(hash/title/planSync/충돌)로 구체화된다.
