# ADR-036: 교회력(전례력) 데이터 모델 · 본기도 정규화 · 우선순위 · 전례색

- 일시: 2026-06-15
- 상태: **제안됨 — 검토 중 (구현 대기).** 사용자 보완 예정. 아직 코드/데이터 미반영.
- 관련: ADR-020(저장소 분리 — 데이터는 `common-bible-data`로 이전 예정), ADR-035(절 부분집합·연속 병합 렌더 = 전례독서 자동 생성 기반), ADR-022(인용·주석), ADR-027(source DSL), 로드맵 Phase 2~4(기도서·교회력·성무일과 — `architecture.md` §11)

## 배경 — 이 ADR이 다루는 범위

성공회 교회력(전례력) 계산기와 그 위에 얹을 **감사성찬례 본기도(Collect)** 데이터의 **모델**을 정한다. 이 ADR은 *데이터 스키마 + 분류 규칙 + 우선순위 체계*를 결정한다.

작업 출발점은 별도 스크래핑 워크스페이스 `eucharist-scrape`(서울교구 블로그에서 수집한 `eucharist-collects.json`, 256개 중복제거 본기도 + 각 본기도의 `used_by` 교회력일 목록 727건)다. 최종적으로 정규화된 데이터는 `common-bible-data`로, 계산 엔진은 `common-bible` 클라이언트 JS로 들어갈 것을 전제한다.

> **구현 현황 (1차, 2026-06-15):** `eucharist-scrape/parse_collects.py`가 본 ADR의 **평면 배열 모델**로 재작성됨. 1차 산출 `eucharist-collects.parsed.json` = **667 entries** (좌표 + 본문 + `_vocab`). 엔진 분류 필드(`rank·precedence·color·priority_group·transferable·outranks_sunday`)는 §9(성인력) 의존이라 **2차/엔진 단계로 보류**. **2차** = 사용자가 타이핑한 축일 본기도(마크다운) 병합 + 분류 필드 채움.

## 결정

### 1. 저장 형태: 계층 트리가 아니라 "자기 좌표를 들고 있는 평면 배열"

`entries`는 **한 레코드 = (교회력일 × 본기도 대체안)**인 평면 배열이다. 각 레코드는 자신의 전례 좌표(`season`·`week`·`type`·`year` 등)를 **필드** 값으로 포함하고, 전례 순서로 정렬된다. 계층(절기→주차→…) 구조는 **저장 형태가 아니라 엔진이 로드 시 만드는 인덱스/뷰**로 파생한다.

**근거 (계산기 관점):**
- 조회 축이 하나가 아니다 — 일반일은 `(season, week, type, weekday, year)`, 고정 축일은 `날짜`, 특수일은 `이름/계산`. 평면 배열이면 로드 시 필요한 인덱스를 여러 개(`Map`) 만들 수 있다. 중첩 트리는 한 가지 조회 순서에 구조를 고정한다.
- 중첩 트리는 `weekday`가 키에 없어 결국 마지막 리스트를 또 필터해야 하고, 축일·재일·특수일(≈25%)이 격자에 안 맞아 `week:"없음"`으로 뭉친다.
- 레코드가 자기 좌표를 들고 다녀 self-describing(“오늘의 본기도” 객체를 그대로 UI·로그·테스트로 넘김). 균일 스키마라 JSON Schema/TS 타입/유닛 테스트가 자명(ADR-013과 정합).
- `week`는 `"1주"` 문자열 대신 **정수**(사전순 함정·범위질의 문제 해소).

### 2. explode 규칙 (256 본기도 → 668 레코드, 교회력일 645종)

- **(교회력일 × 본기도 대체안)마다 한 레코드.** 한 교회력일에 **자유 선택** 대체 본기도가 여럿이면 `collect_no`/`collect_total`로 식별(예: 부활대축일 「다음 세 기도 중 하나」 3개; 주님의 세례 「또는」 2개).
- **제목 끝의 ` N`은 전례독서 트랙 번호다(독서 1 / 독서 2)** — 그 교회력일에 독서 세트가 둘 이상이라 표시된다. **본기도는 트랙과 무관하게 같다**(그런 묶음 82개 중 79개 동일 본문; 달랐던 3건은 모두 오타·오류로 확인·교정 — `연중22주 나해`·`연중26주 가해` 오타, `연중11주 다해`는 두 트랙 본문이 다 잘못이라 소스 교정). → 본기도는 트랙 독립이라 레코드를 하나로 병합한다(`reading_track` 비-null 사례는 현재 데이터에 없음; 필드는 안전상 유지).
  - **독서 연속성 원칙:** 그해 첫 주일에 트랙 1을 택했으면 그해 **모든 주일 독서를 트랙 1로**(2와 섞지 않음) 읽는다 — 봉독의 연속성(반(半)연속 봉독)을 위해서다. 주간 독서도 같은 원칙으로 채택 트랙을 유지한다.
  - 이 **트랙 차원은 본질적으로 독서(lectionary)의 축**이라, 본기도 데이터가 아니라 별도 `eucharist-readings.json` + 교회력 엔진에서 다룬다. 본기도는 (대부분) 트랙 독립이므로 여기서 병합한다. `collect_no`(자유 선택 대체안)와 독서 트랙은 **다른 축**이다.
  - **파싱 규칙(채택):** canonical 제목에서 트랙 번호(` N`)를 떼고, 트랙은 별도 `reading_track` 속성으로 기록한다. 본기도가 트랙 독립이면(병합된 한 레코드가 전 트랙 대표) `reading_track: null`(=전 트랙 공통), 트랙별 본기도가 실제로 다른 경우만 `reading_track: 1|2`로 분리한다.
  - **제목 정규화(채택):** 위 트랙 번호와 함께 **주기 표기**(`(가해)`·`(나해)`·`(다해)`·`(홀수해)`·`(짝수해)`)도 제목에서 제거한다 — 그 정보는 `year` 필드에 있다. 결과적으로 `title`은 **주기·트랙 표기를 뺀 깨끗한 교회력일 이름**(예: `연중10주 화요일`, `대림1주`)이 된다.
  - **별칭(`aliases`):** 동등한 두 이름을 가진 날은 `name`에 대표명(기도서 표제 기준), `aliases`에 나머지 — `성지주일`↔`고난주일`, `성 목요일`↔`성찬제정일`, `성 금요일`↔`주님의 수난일`. 별칭도 검색·조회 키로 동작해야 한다. **별칭이 아닌 슬래시 구분:** ① `주님의 세례 / 연중1주`·`왕이신 그리스도 주일 / 연중34주`처럼 '이름 / 절기좌표'는 좌표(이미 `season`·`week`)이지 별칭이 아니다, ② `성 스테파노(부제, 최초의 순교자)` 괄호는 수식어이지 별칭이 아니다. **주의:** `고난주일`은 **주일(하루)** 별칭이며, 별개의 **주간 이름** `성주간`(=`고난주간`)과 혼동하지 말 것 — 후자는 별칭이 아니라 주간 층위 명칭.
- **복합 주기**(`대림1주 월요일 (나해/다해)`)는 **각 해로 분할·복제**(`나해` 버킷 1건 + `다해` 버킷 1건). 이 때문에 667→**668**.
- **동일 본문 자동 병합**: 같은 교회력일에서 본문이 글자까지 같은 대체안(트랙 포함)은 하나로 합친다. (오타·문장부호만 다른 사실상 중복을 소스에서 교정 후 병합 — 예: `양시을→양심`, `하느님.→하느님,`.)

### 3. 본기도 본문 정제 (필드 `text`)

소스 `text`의 `\n\n` 다중 본기도(15건)는 **의미 단위로 정제**해 한 레코드당 **깨끗한 단일 기도**로 만든다:
- 안내문 제거(`다음 세 기도 중 하나를 드립니다`, `…주간 본기도를 드립니다` 등 — `드립니다` 포함 줄)
- 구분어 `또는` 제거
- 끝의 **공유 송영**(예: 부활대축일 — 본문 3개가 송영 1개를 공유)을 각 기도에 결합

### 4. 값 표기: 영어 정식값 + `_vocab` 한글 라벨

전례 좌표·분류 필드는 **영어 controlled vocabulary**를 정식 값으로 쓰고, 한글 표시 라벨은 레코드마다 반복하지 않고 메타의 **`_vocab` 테이블**에서 단일 정의한다. (혼용 금지 — 한 필드에 한글/영어 섞지 않음.)

```json
"_vocab": {
  "kind":    { "temporal":"이동(시간주기)", "sanctoral":"고정일(성인주기)" },
  "season":  { "advent":"대림","christmas":"성탄","epiphany":"공현",
               "ordinary":"연중","lent":"사순","easter":"부활" },
  "type":    { "sunday":"주일","weekday":"주간","feast":"축일","fast":"재일" },
  "year":    { "A":"가해","B":"나해","C":"다해","I":"홀수해","II":"짝수해" },
  "weekday": { "mon":"월","tue":"화","wed":"수","thu":"목","fri":"금","sat":"토" },
  "rank":    { "principal":"대축일·대재일","privileged_sunday":"절기 주일",
               "major_feast":"주요축일","sunday":"연중 주일",
               "commemoration":"작은 축일·기념일","feria":"평일",
               "regional_festival":"지역 명절" },
  "color":   { "white":"백색","red":"적색","green":"녹색","violet":"자색" }
}
```

**`year` — 두 주기 체계를 한 필드에(레코드당 하나, 상호 배타).** 한 교회력일은 주일·평일·축일 중 하나라 두 주기가 한 레코드에 함께 오지 않으므로 단일 필드로 정규화한다.
- 주일 → `A/B/C`(가·나·다해, 3년 주기)
- 연중 평일 → `I/II`(홀·짝수해, 2년 주기)
- 그 외(대부분 절기 평일·축일·재일·성탄/공현) → `null`. 단 **일부 절기 평일은 해당 본기도가 연도별로 달라 `A/B/C`가 붙는다**(예: 대림1주 월요일) — 소스 표기를 그대로 읽는다.

→ "어느 체계인가"가 `type`만으로 갈리지 않으므로(절기 평일도 `A/B/C` 가능) `sunday_year`/`weekday_year`로 쪼개기보다 단일 필드가 깔끔하다. 값 도메인이 이질적인 비용은 `_vocab` + "type=sunday ⇒ `A/B/C`" 교차 검증으로 완화.

### 5. 구분자 `kind`: temporal / sanctoral

전례력은 날짜 결정 방식이 다른 두 축이 겹쳐 돈다.

- **`temporal`** (시간주기, Proper of Time): **부활절·대림을 축으로 매년 움직이는** 날. 엔진이 *계산*으로 날짜를 구함.
  - 주차 있는 일반일(`대림1주`, `연중10주 화요일`, `사순3주`, `부활5주`)
  - 주차 없는 이름붙은 이동일(`부활대축일`, `성령강림대축일`, `삼위일체`, `승천`, `재의 수요일`, `성지주일`, `성 목/금요일`, `주님의 세례`, `왕이신 그리스도`)
  - **감사주일류**(추수감사·맥추감사·여성선교)와 **ember(사계재일·성소…재)** 도 “어느 달 몇째 주” 또는 “어느 축일 후 X요일”이므로 temporal.
- **`sanctoral`** (성인주기, Proper of Saints): **양력 날짜(MM.DD)에 고정**. 엔진이 *날짜로 조회*. 성인 축일 + 고정 주님 축일(`12.25 성탄`,`1.6 공현`,`8.6 주의 변모`) + 고정 국가일(`3.1 삼일절`,`8.15 광복절`).

같은 변모축일이 `주의 변모 주일`(temporal)과 `8.6 주의 변모`(sanctoral)로 갈리는 것도 이 구분으로 자연스럽게 처리된다.

### 6. 우선순위(precedence) 체계 — 대한성공회 기도서 기반

겹치는 축일은 **품계 비교**로 “누가 그날을 지키고 진 쪽은 이동/기념/생략할지”를 정한다. **성공회는 막힌 축일을 되도록 다른 날로 이동(transfer)** 한다 — 막힌 성인 축일을 그해 그냥 생략하는 로마 가톨릭과 가장 다른 점이다.

품계 사다리(낮을수록 우선):

| prec | rank | priority_group | 해당 | 충돌 시 |
|---|---|---|---|---|
| 1 | `principal` | — | **대축일**(성탄·공현·부활·승천·성령강림·삼위일체·모든성인) + **대재일**(재의수요일·성금요일) | 모든 주일·축일에 우선. 고정(이동 안 함) — 단 **공현·승천·모든성인**만 이동 가능 |
| 2 | `privileged_sunday` | — | 절기 주일(대림·사순·성주간·부활) | 모든 주요축일에 우선 |
| 3 | `major_feast` | **A** | 거룩한이름예수·주의세례·주의봉헌·주의변모·수태고지·왕이신그리스도·성체·성모방문·성모안식·성미카엘·세례자요한 | 연중 주일을 이김. 절기 주일과 겹치면 이동. 단 **거룩한이름예수·주의봉헌·주의변모 3개**는 `outranks_sunday` — **절기 주일까지 이김**(prec 1·2 사이) |
| 4 | `sunday` | — | 연중·공현후 주일 | A에 짐, B·C를 이김 |
| 5 | `major_feast` | **B** | 사도·복음사가·죄없는아기들·성요셉·성막달라마리아·성스테파노·성십자가 | 어느 주일과 겹쳐도 **월요일로 이동**. 대축일·성주간·부활8일과는 겹칠 수 없음 |
| 6 | `major_feast` | **C** | 한국순교자·성니콜라·교구설립일·성당축성일·추수감사주일 | 절기 주일과 겹치면 이동, 어려우면 **기념만** |
| 7 | `commemoration` | — | A/B/C 밖 작은 축일·기념일(안나와요아킴·성모탄신·세례요한참수·별세자의날 등) | 주일에 짐, 막히면 기념/생략 |
| 8 | `feria` | — | 절기 평일 + **소재일**(사순평일·사계재일·금요일 재) | 가장 낮음 |

별도 등급:
- **`regional_festival`**(지역 명절: 설·추석) — 명목상 기도서 C에 있으나 **주요축일·대재일보다도 강한 점유권**(아래 §7 규칙 1·2)이라 별도 rank로 분리한다.

핵심:
- **재일은 한 등급이 아니다.** 재의수요일·성금요일=**대재일(prec 1)**, 사순평일·사계재일·금요일=**소재일(prec 8, 참회 성격 attribute)**. `type:"fast"`는 같아도 `rank`가 갈린다.
- **절기 주일과 연중 주일은 등급이 갈린다**(주요축일 A가 연중 주일은 이기고 절기 주일엔 양보) — 따라서 “모든 주일이 축일보다 우선”은 단일 규칙으로 두지 않는다.

### 7. 충돌·이동 규칙표 (데이터 아님 — 엔진 로직)

precedence로 “누가 그날을 차지하나”는 풀리지만, 진 쪽의 운명과 예외는 규칙표로 둔다:

1. **명절(설·추석) ∧ 주요축일 → 명절 우선, 축일은 기념/이동.**
2. **재의 수요일 ∧ 설 → 재의 수요일을 뒤로 이동.** (대재일도 설엔 양보)
3. **주요축일끼리 겹침 → A > B > C, 진 축일 이동.**
4. 진 축일 이동 목적지: A=다음 빈 날, B=월요일, C=이동 또는 기념.

규칙 1·2가 `regional_festival` 분리의 근거다(명절이 대재일까지 민다).

**모델 함의 — 밀린 독서 보존:** precedence는 그날의 *공식 축일(winner)* 을 정하지만, 엔진은 **밀려난(displaced)·기념(commemorated)·트랙 대안** 관측일을 버리지 말고 보존해야 한다. 그래야 사용자가 "축일에 밀린 기본(평일/연속) 독서"를 놓치지 않고 볼 수 있다. 즉 한 날짜 → **후보 관측일 집합 전체**를 계산해 두고, 공식 1건만 쓰는 것은 *뷰 단계의 선택*이다(§후속 「전례독서 뷰 설정」).

### 8. 전례색 `color` — 명시 저장(절기 기본 + 축일 override)

색은 절기만으로 완전 파생되지 않는다(성령강림은 부활절기 안인데 적색; 순교자·사도는 절기 무관 적색). 따라서 **레코드마다 `color`를 명시 저장**하되 절기 기본값에서 시작해 축일만 덮어쓴다.

| color | 쓰임 |
|---|---|
| `white` 백색(금색) | 성탄·부활·삼위일체·모든성인·주님 축일·비순교 성인·**성 십자가**(십자가의 영광) |
| `red` 적색 | 성령강림·사도/복음사가/순교자·성지주일·성 금요일 |
| `green` 녹색 | 연중(공현 후) |
| `violet` 자색 | 대림·사순·재의 수요일·재일 |

대한성공회는 **자·백·녹·적 4색만** 쓴다 — **장미색 미사용**. 별세자색(흑색 등)은 교회력엔 쓰지 않고 별세자(위령) 성사에서나 쓰인다. (성 십자가는 일반 서방 관례의 적색이 아니라 **십자가의 영광 강조로 백색** — 대한성공회 기준. 색 배정 전체는 기도서로 최종 확인 필요.)

### 9. 음력 계산 (설·추석)

직접 천문 계산을 얹지 않는다. 필요한 건 사실상 **설(음력 1/1)·추석(음력 8/15) 두 날**뿐이므로, **KASI(한국천문연구원) 공식 음양력 변환에서 뽑은 양력 날짜를 연도별 lookup 표**로 둔다(예: 2025–2050). 권위 있고 단순하고 정부 관보와 1:1.

> 한국 음력 ≠ 중국 농력: 방법론(삭망·24절기)은 같지만 **계산 기준 자오선**이 다르다(중국 베이징 UTC+8/120°E, 한국 KST UTC+9/135°E). 삭·절기 순간이 자정 근처면 ~1시간 차가 날짜를 넘겨 **드물게 하루 어긋나거나 윤달 위치가 달라진다.** → 반드시 **한국 기준(KASI)** 으로. 일반 음↔양 변환이 나중에 필요하면 한국 규칙 라이브러리(`korean_lunar_calendar` 등, ~1391–2050)를 쓰고 위 표로 회귀 검증.

`sanctoral`이되 양력 고정이 아닌 설·추석은 `date:null` + `lunar:"1-1"`/`"8-15"`로 두고, 엔진이 `lunar`가 있으면 KASI 표에서 그해 양력 날짜를 끌어온다.

### 10. id

`id`는 **잠정 순번**(전례 순 정렬값). 모델·정렬 확정 후 재지정 예정. 키로 의존하지 않는다.

### 11. 교회력 계산 기준 — 부활절 computus + 대림 앵커 (계산 엔진 전제)

temporal 좌표(`season`·`week`)는 *데이터에 박혀 있지 않다* — **날짜 → 좌표**는 엔진이 두 앵커로 **계산**한다. (상세 엔진은 별도 ADR로 분리하되, 계산 *기준*은 여기서 고정.)

- **부활절(Easter) 규칙:** 교회력 춘분(**3월 21일 고정**) 이후 첫 **교회 보름(Paschal full moon)** 다음 첫 주일. 핵심 — 이 춘분·보름은 **천문값이 아니라 교회력 근사값**(보름은 epact 표 기반)이다. 흔한 정의 '춘분 후 보름 뒤 첫 주일'은 맞지만, **실제 천문으로 계산하면 가끔 어긋난다**(예: 2019년 천문 ↔ 교회력 부활절이 약 한 달 차이). 대한성공회는 **서방(그레고리력) 부활절** — 정교회(율리우스력) computus 아님.
  - **역사:** 부활절 산정 원칙(① 전 교회 일치 ② 주일 ③ 봄 보름 뒤 ④ 유대력 계산과 독립)은 **제1차 니케아 공의회(325년)**에서 정해졌다. 단 '춘분 = 3월 21일 고정'이라는 구체 수치는 공의회 법규가 직접 명문화했다기보다, 니케아 원칙을 따른 **알렉산드리아 교회의 계산 전통**에서 정착한 관행이다(325년 무렵 율리우스력 춘분이 마침 3월 21일).
- **구현(채택):** 위 교회력 규칙을 그대로 인코딩한 **결정론적 Computus 알고리즘**(Anonymous Gregorian / Meeus·Jones·Butcher, ~15줄 정수 연산)을 정본으로. 모든 연도·외부 데이터 0·유지보수 0. **권위 있는 서방 부활절 날짜표로 유닛 테스트 검증**(공표 표 = 교회력 정답이라 회귀 테스트에 적합). 표만 쓰는 방식도 유효하나(설·추석 패턴과 일관) 범위 한계·갱신 부담이 있다 — 부활절 computus는 §9 음력과 달리 *천문이 아니라 닫힌 산식*이라 재구현이 안전하다.
- **부활절에서 파생되는 이동 축일**(부활주일 `E` 기준 오프셋):
  - 재의 수요일 `E−46` · 사순1주일 `E−42` · 성지(고난)주일 `E−7` · 성 목요일 `E−3` · 성 금요일 `E−2`
  - 승천 `E+39`(목) · 성령강림 `E+49` · 삼위일체 `E+56` · (그리스도의 성체 `E+60`)
- **대림·성탄 앵커(고정일 → 가까운 주일):** 대림1주일 = 성탄(12.25) **4주 전 주일**(=11.30에 가장 가까운 주일). 성탄 12.25·공현 1.6 고정. 주님의 세례 = 공현 후 주일.
- **연중(Ordinary) 길이는 부활절 위치에 따라 변동한다.** 전반부(주님의 세례 다음날 ~ 재의 수요일 전)와 후반부(성령강림/삼위일체 후 ~ 대림 전)의 주차 수가 해마다 다르다. 후반부 주차 번호는 **끝(왕이신 그리스도 주일 = 연중34주, 대림 직전 주일)에서 역산**해 매긴다. → 엔진이 풀어야 할 핵심 난점.

> 함의: 같은 `season:"ordinary", week:N`이라도 *실제 날짜*는 해마다 다르다. 그래서 데이터는 좌표만 들고, 날짜 매핑은 엔진이 부활절·대림 계산으로 런타임 산출한다.

## 검토한 대안

### 저장 형태
- **A. 평면 배열 + 좌표 필드 (채택)** — 다축 조회·균일 스키마·self-describing. 계층은 런타임 인덱스로.
- **B. 4계층 중첩 트리(season→week→type→year)** — 보기엔 깔끔하나 단일 조회축 고정, weekday 미반영, 축일/특수일이 격자 밖, 데이터-as-키 안티패턴. (현 `parse_collects.py` 형태 — 기각)

### 값 표기
- **A. 영어 값 + `_vocab` 한글 라벨 (채택)** — 코드 분기·타입·검증 친화, 인코딩 함정 회피, 라벨 단일 출처.
- **B. 한글 값** — 한글-우선엔 자연스럽지만 `switch`·타입·키 안정성에 불리.
- **C. 인라인 `*_label`** — 반복·동기화 부담.

### 우선순위 표현
- **A. precedence 정수 + priority_group + 규칙표 (채택)** — 기도서가 A/B/C로 규칙을 정의하므로 그 구조를 그대로 인코딩.
- **B. 단일 “주일 > 축일” 규칙** — 기도서의 A 특례(연중 주일을 이기는 축일)를 표현 못 함. 기각.

### 전례색
- **A. 명시 저장 + 절기 기본 override (채택)** — 성령강림 적색·순교자 적색 등 파생 불가 케이스 안전.
- **B. season에서 순수 파생** — override 많아 취약. 기각.

### 음력
- **A. KASI 양력 lookup 표 (채택)** — 필요한 날 2개뿐, 권위·단순·검증 가능.
- **B. 음력 천문 계산기 내장** — 과설계·오류 위험.

## 성인력 보완 영역 — 영명축일·성당축성일·성인 공통 (제안, 도메인 확인 필요)

현재 sanctoral 모델은 **스크랩된 본기도가 있는 성인 축일 일부**만 담는다. 전체 성인력(성인력 권위 출처: 성공회 기도서)과 그 운용은 아직 미반영이다. 빠진 세 갈래:

### (1) 영명축일·성당축성일 — 지역(본당)·개인 축일

> **성공회적 신학 전제:** 성공회는 성인을 *전구(intercession) 대상*이 아니라 *본받을 모범·감사 대상*으로 본다(39개조 신조 제22조 — 성인에게 드리는 기도 배격). 따라서 RC와 달리 **개인 명명일(세례명) 영명축일은 비중이 낮은 선택 기능**이고, **본령은 본당 차원**(주보성인 축일·성당축성일 — 공동체 잔치이자 공적 전례 등급 격상)이다. 아래 '본당=등급 격상 / 개인=알림' 효과 구분이 이 차이를 그대로 반영한다.

- **영명축일**(주보성인 축일)·**성당축성일**은 *그 본당에서만* 높은 등급으로 지킨다. 같은 성인의 날이 보편 달력엔 작은 기념일이어도, 그 성인을 주보로 모신 본당에선 주요축일급.
- 즉 한 날의 `rank`가 **보는 주체(본당/개인)에 따라 달라진다** → 단일 전역 rank로 표현 불가.
- **제안:** 보편 달력 위에 **지역/개인 축일 오버레이**를 둔다. 보편 레코드는 그대로 두고, 본당 설정(주보성인·축성일)이 해당 날의 등급을 격상. 레코드/엔진에 `scope`(`universal` | `local`) 개념 도입.
- 우선순위·이동: 영명축일·성당축성일은 보통 주요축일급이며, 절기 주일·대축일과 겹치면 §7 이동 규칙 적용.
- **설정 기능(제안):** 설정에서 **본당 영명축일(주보성인)·성당축성일**과 **독자(개인) 영명축일(세례명 성인)**을 지정. 효과가 다르다 —
  - 본당 영명축일·성당축성일 → *공적 전례의 등급 격상*(그날 그 본당 감사성찬례가 주보 본기도 사용).
  - 개인 영명축일 → *개인 표시/알림*(공적 전례는 불변).
- **개인 영명축일 폴백:** 세례명에 해당하는 성인이 (데이터에 없거나 고유 축일이 없어) 매칭 안 되면 **「모든 성인의 날」(11.1)** 을 개인 영명축일로 둔다 — 모든 성인의 날이 개별 기념 안 되는 성인 전체의 보편 기념이라 전통적으로도 부합. 이미 대축일이라 등급 충돌 없음.
- **다중 성인:** 한 세례명에 성인이 여럿(여러 요한·마리아 등)이면 설정에서 *어느 성인(=어느 날)*인지 선택. 폴백은 진짜 매칭이 없을 때만.

### (2) 성인 분류 속성 `sanctoral_class` — 색·등급·공통 선택의 입력

- 성인 축일의 `color`·`priority_group`·공통 선택이 **성인 분류**에서 파생된다(순교자→적색, 사도→`B`, 비순교 성인→백색 등).
- **제안 필드:** `sanctoral_class`(`martyr` | `apostle` | `evangelist` | `bishop` | `pastor` | `virgin` | …). 색·rank·Common 자동 선택의 입력값.

### (3) 성인 공통 본기도(Common of Saints) — 폴백

- 고유 본기도가 없는 성인은 분류별 **공통 본기도**(순교자 공통·사도 공통·목자 공통…)를 쓴다. 스크랩 데이터엔 고유본만 있어 나머지는 공통으로 채워야 한다.
- **제안:** 별도 `commons` 테이블(분류 → 공통 본기도). 성인 레코드가 고유본 없으면 `sanctoral_class`로 공통을 참조.

**도메인 확인 필요:**
- 영명축일은 **본당 단위**인가, **개인(세례명) 단위**도 다루나(개인 알림용)?
- ✅ 대한성공회 성인력은 **공통 기도문(Common)** 체계를 쓴다(사용자 확인). 남은 것: 분류(`sanctoral_class`) 목록 확정.
- 영명축일·성당축성일의 정확한 품계(주요축일 A/B/C 중 어디)와 이동 규칙.
- 전체 성인력의 권위 출처·범위(어느 기도서/성인력 판본).

## 미결 사항 (사용자 보완 대상)

1. ✅ **품계 8단계 사다리** — 채택(prec 7 작은 축일 별도 등급 포함).
2. ✅ **명절 `regional_festival` 분리** — 동의.
3. ✅ **A 특례 3개 `outranks_sunday`** — `거룩한이름예수·주의봉헌·주의변모`는 **절기 주일까지** 이긴다(prec 1·2 사이; §3·§6 반영).
4. ✅ **전례색 = 자·백·녹·적 4색만** — **장미색·별세자색 미사용**(별세자색은 위령 성사용; §8 반영).
5. ◐ **분류 잔여** — **성 금요일 = 재일(대재일)** 확정(세계 성공회도 동일: rank=주요 성일, character=단식 → `type:fast`+`rank:principal`). 성 목요일=축일·성령강림 season=부활·삼위일체 season=연중은 현 ADR 유지(이견 시 조정).
6. ✅ **연중11주 다해** — 트랙 1·2 본문이 둘 다 오류였고 소스 교정됨([10781](https://seoul.anglican.kr/archives/10781)·[10782](https://seoul.anglican.kr/archives/10782)). 트랙 변이가 아니라 오류 → **본기도는 완전히 트랙 독립**(§2 반영). 로컬 스크랩 재수집 필요. (전례독서 트랙·연속성·태그 스킴은 readings/캘린더 ADR 소관 — 좌표 태그 `#가해_연중1주일` 등은 §4 vocabulary와 1:1, 분류 태그 `#대축일`·`#성인축일` 등은 §6 `rank`/`kind`와 정합·「공통(Common) 독서」 표지.)
7. ✅ **데이터 거처 = `common-bible-data`** — 세부 경로·독서/시편 통합 구조는 구현 시.
8. ◐ **본기도 본문 검증** — 텍스트가 서울교구 블로그 스크랩이라 공식 기도서와 다를 수 있음(이미 오타 다수 발견·교정). 앱이 *공식 본기도*로 제시하므로 **공식 기도서 대조**를 데이터 품질 과제로. (← 앞서 "본문 출처"를 물은 이유.)
9. ◐ **성인력 보완 예정** — `scope`(영명축일·축성일 오버레이)·`sanctoral_class`·**공통 기도문(Common — 존재 확인됨)**·전체 성인력 권위 출처. 현재 스크랩 subset 범위 밖.
