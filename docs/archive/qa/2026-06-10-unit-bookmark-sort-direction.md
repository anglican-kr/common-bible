# 유닛 테스트 보고서: 북마크 정렬 방향(오름/내림) 선택

**날짜:** 2026-06-10
**범위:** `js/app/bookmark-core.js` — 북마크 목록의 정렬 기준(제목·날짜 등)에 더해, **오름차순/내림차순 방향**을 직접 고를 수 있게 했다.
**관련 ADR:** ADR-010(북마크 — 정렬), ADR-030 후속⁷(⋯ 메뉴 가로 스크롤)
**관련 PR:** (작성 중, 브랜치 `feat/bookmark-sort-direction`)

## 한 줄 요약

북마크 목록 우측 위 ⋯ 메뉴에서 **정렬 기준을 고르는 칸 아래에 "오름차순 / 내림차순" 한 쌍이 새로 생겼다.** 예전엔 기준마다 방향이 정해져 있어(제목은 가나다순, 날짜는 늘 최신순) 거꾸로 볼 수 없었는데, 이제 같은 기준이라도 방향을 뒤집을 수 있다.

## 무엇이 바뀌었나 (사용자 입장)

- **위치:** ⋯ 메뉴를 열면 **새 폴더·내보내기·가져오기·선택**(맨 위) → 가는 선 → **정렬 기준**(제목·직접 정렬·추가된 날짜·최근에 본 날짜·수정한 날짜) → 가는 선 → **오름차순 / 내림차순**(맨 아래) 순서다(아이폰 음악 앱과 같은 짜임새).
- **기준별로 따로 기억:** "제목"을 거꾸로(내림차순) 봤다가 "추가된 날짜"로 바꾸면, 날짜는 날짜대로 마지막에 본 방향을 보여준다. 기준을 오갈 때마다 방향이 뒤섞이지 않는다.
- **처음 그대로:** 한 번도 방향을 건드리지 않았다면 예전과 **똑같은 순서**로 보인다 — 제목은 가나다순, 날짜 3종은 최신순이 기본값이다. 그러니 이번 변경으로 기존 사용자가 놀랄 일은 없다.
- **"직접 정렬"에는 방향이 없다:** 손으로 끌어 옮긴 순서(직접 정렬)는 오름/내림 개념이 없으므로, 그 모드일 땐 아래 두 줄이 **흐리게(누를 수 없게)** 표시된다.
- **저장 위치:** 방향 설정은 정렬 기준과 마찬가지로 **이 기기에만** 저장되고(브라우저 저장소), 구글 드라이브로 동기화되지 않는다 — 북마크 내용이 아니라 "보는 방식" 설정이기 때문이다.

## 무엇이 더 안전/깔끔해졌는가

정렬 비교 로직을 **"오름차순 기준 하나"로 단순화**하고, 내림차순은 그 결과의 부호만 뒤집는 방식으로 바꿨다(기준마다 정·역방향 코드를 따로 두지 않음). 폴더를 항상 앞에 모으는 규칙은 그대로다.

새로 추가한 자동 점검(`bookmark.js` 단위 **169건**, 전체 회귀 **728건** 통과, 타입 검사 0건 경고):

### 방향 기본값 — 2건
기준별 자연 기본값(제목 오름·날짜 내림)을 돌려준다 / 방향이 없는 모드(직접 정렬)나 모르는 값은 안전하게 "오름"으로 떨어진다.

### 방향 저장·복원 — 3건
저장된 방향을 읽되 깨진 값은 기본값으로 / 기준마다 **따로** 저장돼 서로 안 건드림 / "직접 정렬"과 잘못된 방향 값은 **저장 거부**(엉뚱한 기록 방지).

### 실제 정렬 뒤집힘 — 2건
"추가된 날짜 + 오름차순" → 오래된 것부터 / "제목 + 내림차순" → ㅎ→ㄱ, 그래도 **폴더는 여전히 맨 앞**.

## 자동 점검이 닿지 않는 부분 (수동/e2e 확인)

메뉴 안 두 번째 라디오 그룹의 **체크 표시 갱신**(기준을 바꾸면 그 기준의 방향에 체크가 옮겨감), **직접 정렬일 때 흐리게 비활성**, 방향을 누르면 목록이 다시 그려지는 흐름은 브라우저에서 확인했다(Playwright, 모바일 화면): 직접 정렬에선 두 줄 비활성, "추가된 날짜" 선택 시 내림차순에 기본 체크, 오름차순을 누르면 `{"created":"asc"}`로 저장되고 재오픈 시 체크가 옮겨짐, 페이지 오류 0건. 또한 줄이 11개로 늘어난 메뉴가 **가로 모드에서 스크롤로 모두 닿는지**(ADR-030 후속⁷)도 함께 확인했다.
