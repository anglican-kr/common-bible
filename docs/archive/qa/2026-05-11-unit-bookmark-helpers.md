# 유닛 테스트 보고서: 북마크 — 링크 생성·활성 표시·내보내기/가져오기

**날짜:** 2026-05-11
**범위:** `js/app/bookmark.js` — 북마크 항목의 URL 만들기, 현재 페이지와 일치하는 북마크 표시하기, 북마크 파일 내보내기/가져오기 검증·병합
**관련 PR:** (작성 중)

## 한 줄 요약

각 북마크가 어떤 주소로 연결되는지 정확히 만들어내고, 지금 보고 있는 페이지에 해당하는 북마크를 자동으로 강조 표시하고, 다른 기기에서 받은 북마크 파일을 안전하게 검증·병합할 수 있는지 자동으로 확인하기 시작했다.

## 이 변경으로 무엇이 더 안전해졌는가

자동 검증 케이스 수가 **406건 → 440건**으로 늘었다 (+34). 북마크 모듈 단독으로는 70 → 104.

이번에 자동으로 점검하게 된 동작:

### 북마크 링크 만들기

각 북마크가 클릭됐을 때 이동할 주소를 정확히 생성:

- 장 전체를 가리키는 북마크 → `/창세기책ID/1`처럼 깔끔하게
- 특정 절을 가리키는 북마크 → `/마태책ID/5/3-12`처럼 절 범위까지
- 여러 절을 묶은 북마크(예: 1절·3-5절·9절) → `/로마서책ID/8/1,3-5,9` 그대로 보존
- 시편처럼 단위가 "편"인 경우든, 일반 책이든 똑같이 안전

### 현재 페이지와 매칭되는 북마크 자동 강조

지금 보고 있는 페이지가 어떤 북마크에 해당하는지 알아내서, 북마크 목록에서 그 항목을 시각적으로 구별:

- 장 전체 페이지를 보고 있으면 "장 전체" 북마크가 활성으로 표시
- 절 범위 페이지를 보고 있으면 정확히 그 범위 북마크만 활성 (장 전체 북마크는 활성 안 됨 — 다른 자원이라고 판단)
- 다른 장·다른 책으로 가면 즉시 비활성으로 전환
- **폴더 안에 활성 북마크가 있는지 추적** — 폴더가 닫혀 있어도 안에 활성 북마크가 있으면 폴더 자체도 강조될 수 있도록 깊은 폴더 구조(여러 단계 중첩)에서도 정확히 탐지
- 빈 폴더, 자식이 비어 있는 폴더, 자식 정보 자체가 빠진 폴더에서도 오류 없이 false 반환

### 북마크 파일 검증 (가져오기 전)

다른 기기·다른 사용자에게서 받은 .json 파일이 정말 북마크 파일인지 안전하게 확인:

- null·undefined·문자열·숫자·boolean 같은 잘못된 입력은 모두 거절
- 객체이긴 한데 `bookmarks` 항목이 없거나 배열이 아니면 거절
- 정상적인 구조 (`{ bookmarks: [...] }`)이면 허용 (빈 배열도 유효)
- 깨진 JSON 파일을 그대로 적용해서 기존 북마크가 사라지는 사고 방지

### 북마크 병합 (기존 + 들어오는)

두 북마크 묶음을 합칠 때 중복 없이, 기존 데이터 우선:

- 같은 ID가 있으면 **기존 것을 유지하고 들어오는 것은 버림** — 라벨이나 메모가 우연히 같은 ID로 바뀌어서 덮어쓰는 사고 없음
- 새로운 ID들은 모두 추가
- 폴더 안의 자식들도 ID로 정확히 거름 — 폴더 안에 동일 ID 북마크가 있으면 그것만 제외하고 나머지 자식은 그대로
- 폴더 자체가 중복이면 폴더 전체(자식 포함) 통째로 건너뜀 — 같은 폴더에 들어 있던 새 항목들은 추가 안 됨 (사용자가 폴더 단위 충돌을 명확히 해결할 수 있도록)
- 여러 단계로 중첩된 폴더(폴더 안의 폴더 안의 폴더)에서도 각 단계에서 정확히 필터링
- 자식 정보가 아예 없는 폴더가 들어와도 빈 자식 배열로 안전하게 처리

### 북마크 개수 세기

가져오기 모달에서 "북마크 N개를 병합하시겠어요?"를 정확히 보여주기 위해:

- 폴더 자체는 카운트에 포함 안 함 (사용자가 인식하는 "북마크 개수"는 실제 북마크 항목들의 합)
- 폴더 안 깊숙이 있어도 모두 합산
- 폴더가 비어 있거나 자식 정보가 없어도 오류 없이 0으로 간주

## 발견된 이슈

없음. 위 모든 동작이 의도한 대로 동작함을 확인.

작은 리팩토링: `_isActiveBookmark`와 `_hasActiveDescendant`에 옵션 인자(`pathname`)를 추가. 호출 측은 인자 없이 호출하면 기존과 동일하게 동작하고, 테스트는 명시적으로 페이지 경로를 넘겨서 검증할 수 있게 되어 검증 가능성이 크게 향상.

신규 BEGIN/END 마커 영역 (`js/app/bookmark.js`):
- `BOOKMARK_HREF` — `_bookmarkHref` 순수 URL 빌더
- `BOOKMARK_ACTIVE` — `_renderPathname` + `_isActiveBookmark` + `_hasActiveDescendant`
- `IMPORT_EXPORT` — `_validateImportData` + `_mergeBookmarkStores` + `_countBookmarks`

코드 리팩토링: `_isActiveBookmark(bm, pathname = _renderPathname)` / `_hasActiveDescendant(folder, pathname = _renderPathname)` — 옵셔널 두 번째 인자 추가. 기존 호출자는 인자 없이 호출하므로 default 값(`_renderPathname`)을 그대로 사용, 동작 변경 0. 테스트는 명시적 pathname을 넘겨 module-state 변경 없이 검증.

테스트 패턴: HREF 블록은 의존 없음. ACTIVE 블록은 HREF 블록을 concat해서 같은 vm 컨텍스트에 로드 (의존 해결). IMPORT_EXPORT는 의존 없음. 모든 함수가 순수 → 매우 견고한 테스트.

DOM 헤비 영역(drawer/modal/tree render/drag) 의도적 스킵 (ADR-013 dual-track 기조).
